# Oracle — 오라클 기반 검증 (WHAT IS CORRECT)

> 스펙을 잘 써도 "생성된 코드가 **맞는지**"는 별개다. 그 판정 기준이 **오라클(Oracle)** 이다.
> 이 문서는 `lunch-recommender`의 **3층 오라클**을 정의한다. `spec.md`(무엇)·`plan.md`(어떻게)에 이어, "**무엇이 정답인가**"를 담당한다.
> 개념 배경: 스터디 3회차 — 오라클 프레임워크(바이너리·명세·도메인 오라클, 콜드 스타트).

## 0. 왜 필요한가

`spec.md §6`을 "검증 oracle"로 쓴다는 원칙(CLAUDE.md)은 사실 **명세 오라클 하나만** 가동한 상태였다. 하지만:

- 스펙(AC)은 **정해둔 시나리오**만 판정한다. AC에 없는 입력·경로에서 이상한 값이 나와도 못 잡는다 → **도메인 오라클**이 필요.
- 리팩터·성능개선처럼 "동작은 그대로여야 하는" 변경은 스펙이 아니라 **이전 동작 자체**가 정답이다 → **바이너리 오라클**이 필요.

AI가 코드를 순식간에 뽑는 시대에 남는 유일한 레버는 "**이 결과가 맞는가를 뭘로 판정하나**"다. 도구(Playwright 등)를 고르기 전에 **"그래서 뭘 비교할 건데?"** 를 먼저 답한다.

## 1. 이 프로젝트의 3층 오라클

### ① 명세 오라클 — "스펙대로인가?" (현재 코드 없이도 바로 대비 가능한 판정 기준)

정답 = `spec.md §6 성공기준` + `plan.md 테스트 관점`. 새 기능은 기존 동작이 답을 못 주므로 사람이 쓴 명세가 정답이다. 판정 기준은 확립됐으나 **통과시킬 코드는 Phase 5** — 아래 AC는 아직 전부 미충족(코드 부재)이다.

체크 가능한 AC — **`spec §6`의 전량 미러**(원천은 `spec.md §6`, 스펙이 바뀌면 이 목록도 동기화한다. 두 번째 진실 원천을 만들지 않기 위함):

- [ ] 한 번의 클릭으로 반경 내 **최근과 겹치지 않는** 식당 1곳을 지도·거리·카카오맵 링크와 함께 제시
- [ ] 연속으로 눌러도 **최근 N회(= `RECENT_LIMIT`회)**는 서로 다른 곳
- [ ] 추천 식당이 **도보 15분(직선 ≈1km) 이내**
- [ ] **야간 전용 업종(술집·바 등) 제외**
- [ ] **대표 메뉴 힌트**가 업종 기반으로 함께 표시(추정임을 표기)
- [ ] 필터 후 후보가 없어도 오류 없이 **안내 + 대안(반경 확대/이력 초기화)**

> 품질 장치(하네스): 요구가 모호/누락이면 코드에서 추측하지 말고 `spec.md`를 **보완 후 재생성**한다(= DOC_GAP / NEEDS_INPUT 취지). CLAUDE.md 운영원칙과 동일.

### ② 도메인 오라클 — "상식적으로 맞나?" (스펙에 안 써도 당연한 규칙)

시나리오(AC)로 정의하지 않은 입력·경로에서도 **항상 성립해야 하는 불변식**. 위반 시 즉시 FAIL. 대부분 이미 `constitution.md`·`spec.md`에 흩어져 있던 "당연한" 것들을 **판정 가능한 규칙**으로 모은 것이다.

| # | 도메인 규칙(불변식) | 근거 |
|---|---|---|
| D1 | 추천 식당은 항상 CENTER로부터 `distance ≤ RADIUS`(도보 15분). 반경 밖이 나오면 비정상 | spec §4 거리 제한 |
| D2 | 추천 대상은 FD6(음식점)이고 **야간 전용 업종**(술집/호프/바/포차/이자카야/야식…)이 아니다 | spec §4 점심영업 근사 |
| D3 | **가용 후보 수 > `RECENT_LIMIT`인 한**, 최근 `RECENT_LIMIT`개는 서로 다르다(중복 추천 = 비정상). 후보가 그보다 적어 순환하는 건 정상 — 이 조건을 판정기에 포함해 오탐을 막는다. (가용 후보 = 반경·점심영업 필터 통과한 서로 다른 `place.id` 총수, 이력 제외 전) | spec §4, plan 추천로직 |
| D4 | 표시되는 모든 값은 Kakao **실데이터 근거**. 없는 식당·메뉴·영업시간을 지어내지 않는다. 근사값(거리·점심영업)은 "근사/추정"으로 표기 | constitution 3, 창작금지 |
| D5 | 개인정보(이력·취향)는 **localStorage에만**. 네트워크로 나가지 않는다. 로그인·서버저장 없음 | constitution 2 |
| D6 | 클라이언트 노출 키는 **도메인 제한 JS 앱키만**. REST 키·시크릿은 코드/레포에 없다 | constitution 4 |
| D7 | 필터(반경·점심영업·이력) 후 후보가 0이어도 **크래시 없이 안내**한다 | spec §4 후보 0 처리 |
| D8 | **격자 분할 검색의 합집합이 반경을 빠짐없이 덮는다** — Kakao 45개 상한으로 후보가 조용히 절단되지 않는다(절단되면 랜덤 대표성 훼손). 탐지 술어: **어떤 타일도 `count == 45 && is_end == false`가 아니다**(cap 도달+더 있음 = 절단) | plan 후보수집 상한 |
| D9 | **지오코딩 실패 시 임의 CENTER 좌표를 만들지 않는다** — 안내 후 중단(중심 좌표 날조 금지) | plan 데이터소스, 창작금지 |

**검증 방식** — 도메인 오라클은 "시나리오 없는 경로"를 잡는 게 핵심이다. 고정 시드/모의 Kakao 응답에 대해 **대량·랜덤 입력**(여러 좌표·이력 상태·검색결과)을 넣고, 위 D1~D9가 깨지는 산출이 하나라도 나오는지 자동 점검한다. (AI 판단이 아니라 기계적 불변식이라 오탐이 적다.)

### ③ 바이너리 오라클 — "이전과 똑같이 도나?" (아직 없음 — 콜드 스타트 중)

정답 = **확정된 기존 동작(golden)**. 잘 돌아가는 구현 자체가 수만 건의 TC가 된다.

- **현재 부재**: Phase 5(구현)가 아직 없어 비교할 바이너리가 없다.
- **생성 계획**: Phase 5 첫 구현이 **수용**되는 순간(아래 §2), 대표 입출력을 **golden 스냅샷**으로 박제한다. 이후 리팩터·성능개선은 코드를 읽지 않고 golden과 **diff만** 본다(같으면 PASS = regression).
- **전제조건(결정성)**: 추천은 난수(랜덤)라 golden 재현엔 **결정성**이 필요하다. 표준 `Math.random()`은 **시드를 못 넣으므로**, 난수원을 **주입 가능**하게 설계해 검증 시 고정 시드 PRNG를 주입한다(→ `plan.md` 추천 로직의 설계 불변식). 이게 없으면 golden 스냅샷은 매번 달라져 재현 불가.
- **golden 후보(수용 시 박제)**: 고정 시드 PRNG + 모의 Kakao 응답(결정적)에 대한 〈추천 결과 · 메뉴 힌트 · 거리 표기 **문자열**〉 스냅샷(= 로직 출력). 라이브 Kakao 응답은 비결정적이라 golden에 쓰지 않는다(모의 고정). **주의**: golden은 로직 출력 문자열이지 렌더된 화면(카드 레이아웃·다크모드·DOM)이 아니다 — 시각 회귀가 필요하면 별도 스크린샷/DOM 스냅샷 오라클을 둔다(§4).

## 2. 콜드 스타트 — 지금 이 프로젝트의 위치

새 프로젝트라 **비교할 기존 바이너리가 없다 = 콜드 스타트**. 이때 명세 오라클만이 유일한 정답이다.

- 스펙은 "**의도**"를 정하지 "**형태**"를 정하지 않는다. "결과 카드에 식당을 보여준다"는 스펙에서 카드 레이아웃·랜덤 구현·격자 타일 크기는 여러 가지가 가능하다 — 같은 스펙에서 다른 구현이 나오는 건 버그가 아니라 자연스러운 속성.
- **수용의 순간** = 여러 가능성 중 하나를 "이걸로 가자"고 정하는 때. 이 프로젝트에선 **① `spec §6` AC 통과 + ② 실기기(모바일) 확인 + ③ GitHub Pages 배포 확인**이 그 기준이다. 이 순간 세 가지가 동시에 일어난다: 스펙 충족 인정 · 다른 구현 가능성 포기 · **바이너리 오라클(golden) 탄생**.
- **사람 눈 판정은 이 최초 수용 1회에 한해 허용**된다(②의 실기기 확인). 그 이후 모든 회귀·재검증은 golden/AC/도메인 규칙 대비(자동)로만 판정하고 "눈으로 OK"를 금한다 — 이 문장이 CLAUDE.md·sdd-cycle·§4가 인용하는 정본이다.

```
[콜드 스타트 · 명세 오라클]                 ← 지금 여기
  spec 이 가능한 구현 범위를 정의
    → 구현(Phase 5)이 하나를 선택
    → 수용(AC 통과 + 실기기 + 배포)
    → golden 박제 = 바이너리 오라클 생성 ──┐
                                            ▼
[정상 운영 · 바이너리 오라클]
  이후 모든 변경을 golden 과 비교(regression)
  의도적 변경이 필요하면 → Delta 를 spec/규칙에 반영 → 재생성 → 재확정(golden 승격)
```

> **Delta** = "이번에 정답이 어떻게 바뀌나". `기존 정답 + Delta = 새 정답`. 예) 메뉴 힌트 매핑을 바꾸면 golden을 손으로 고치지 말고 `spec/plan`(또는 config 규칙)을 고쳐 재생성 → 새 golden 승격.

## 3. 오라클 성숙도 (자가 진단)

> ⚠️ 사내 가이드의 사다리(`L1=바이너리만`)는 **레거시가 먼저 있는 브라운필드** 전제다(바이너리부터 확보). 이 레포는 **명세가 먼저인 SDD 그린필드**라, 그대로 쓰면 "명세는 있고 바이너리는 없는" 현 상태를 표현할 칸이 없다. 그래서 아래처럼 **명세-우선**으로 재정의해 쓴다.

| Level | 상태 (명세-우선 재정의) | 이 프로젝트 |
|---|---|---|
| L0 | 오라클 없음(사람 눈만) | — |
| L1 | **명세 오라클만** 실동작 | **현재** |
| L2 | 명세 + 바이너리 | 목표(Phase 5 첫 구현 **수용** 시) |
| L3 | 3층(+도메인 자동점검) + 자동 진화 | 지향 |

**현재: L1.** 명세 오라클은 판정 기준으로 확립(문서), 단 통과시킬 코드는 Phase 5. 성숙도는 **"실동작하는 오라클"** 기준으로 매긴다: 도메인 오라클은 규칙(D1~D9)을 **정의**했으나 판정하는 자동 점검이 없어(Phase 5) **레벨을 올리지 않고 L3의 준비 상태**다. 바이너리 오라클은 **콜드 스타트**(구현 수용 후 생성). 즉 수용 시 명세+바이너리가 서면 L2, 도메인 자동점검·자동 진화까지 붙으면 L3. 성숙도는 "테스트를 얼마나 짰나"가 아니라 "**정답을 얼마나 체계적으로 정의·자동 갱신하나**"로 본다.
> 주의: 여기 `L1~L3`은 위 ⚠️대로 **명세-우선 재정의**라, 사내 가이드의 `L1(바이너리만)`과 같은 숫자가 다른 뜻이다. 팀 간 성숙도 비교 시 "명세-우선 기준"임을 함께 밝힐 것.

## 4. 변경 시 — 어떤 오라클로 판정하나 (실무 체크리스트)

| 변경 유형 | 1차 오라클 | 보조 |
|---|---|---|
| 기존 로직 수정 (동작 안 바뀜, 리팩터) | 바이너리(golden regression) | — |
| 기존 로직 수정 (동작 바뀜) | 바이너리 + 명세 | 도메인 |
| 신규 기능 | 명세 | 도메인 |
| UI/UX 변경 | 명세 + 도메인 | (시각) 스크린샷/DOM 스냅샷 오라클 — 도입 시 |

> golden(로직 출력 문자열)은 렌더링을 못 잡으므로 UI/UX 변경엔 바이너리 대신 시각 스냅샷 오라클이 필요하다(§1③ 주의).

PR·리뷰 때 한 줄 질문: **"이 변경의 정답은 뭘로 정해져 있나?"** (golden? AC? 도메인 규칙?) — **회귀·재검증**은 evidence 없이 "눈으로 OK" 금지. 단 **최초 수용**(§2)은 사람 판단 1회 허용(실기기 확인).
