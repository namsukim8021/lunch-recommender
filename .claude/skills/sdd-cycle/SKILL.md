# sdd-cycle — 기능 한 사이클(스펙→생성→검증→보완) 표준 절차

목적: 이 레포에서 기능을 구현·검증할 때 **"코드 직접 수정 금지, 스펙 대비 검증"** SDD 루프를 표준화한다(Harness의 **Verify·Correct** 축). 규약은 [/CLAUDE.md](../../../CLAUDE.md) 참조.

## 언제 사용
- `lunch-recommender`에서 새 기능/변경을 구현할 때, 또는 생성물을 검증할 때.

## 절차
1. **SPEC 확인** — `docs/spec.md`(무엇)·`docs/plan.md`(어떻게)에서 대상 기능의 요구·성공기준·설계를 읽는다. 요구가 모호/누락이면 **먼저 문서를 보완**(문서 우선).
2. **GENERATE** — `plan.md` 설계대로 코드를 생성한다(바닐라 JS, `config.js` 규약 준수, 정적·무료·키안전 유지).
3. **VERIFY (oracle = 스펙)** — 생성물을 `spec.md §6 성공기준` + `plan.md 테스트 관점`과 **1:1 대조**한다. 최소 점검:
   - 후보가 **도보 15분(직선 ≈1km) 이내**인가 / **야간 업종(술집·바 등) 제외**되는가
   - **최근 N회 안 겹치는가** / **후보 0 UX**(안내+반경확대/이력초기화)
   - **메뉴 힌트**가 category 리프/매핑으로 도출되는가 / **거리 직선·도보근사 정직표기**
   - **지오코딩 실패** 시 임의 좌표 없이 안내하는가 / **45개 상한 격자 완화**로 전수에 가까운가
   - 키는 도메인 제한 JS 앱키만 쓰는가(시크릿 미노출)
4. **CORRECT** — 불일치가 있으면 **코드를 직접 고치지 말고**:
   - 요구 자체가 모호/누락 → `docs/spec.md` 보완 후 2로 복귀(재생성).
   - 설계만 어긋남 → `docs/plan.md` 보완 후 재생성.
5. **RECORD** — 요청 프롬프트는 `PROMPTS.md`에 누적. 커밋 트레일러는 CLAUDE.md 규칙 준수.

## 금지
- 스펙에 없는 동작을 코드에서 임의 추가/수정(스펙 없는 변경 금지).
- 생성물을 스펙 대비 없이 "눈으로 OK" 판정.
- 시크릿(REST 키 등)·사내 기밀을 코드/문서/커밋에 노출.
